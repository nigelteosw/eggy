# Eggy MVP Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the approved single-user Eggy daemon and CLI with Telegram, DeepSeek, scheduling, Codex, GitHub, and Google Calendar integrations behind provider-neutral Go ports.

**Architecture:** A ports-and-adapters modular monolith keeps kernel event dispatch, policy, tasks, and assistant orchestration independent of providers. The daemon composition root loads YAML and environment secrets, constructs file-backed adapters, and exposes the five HTTP endpoints; the CLI operates on the same configuration and state files.

**Tech Stack:** Go 1.26, Go standard library HTTP/JSON/crypto/process packages, `gopkg.in/yaml.v3`, Docker, Git, Codex CLI, Telegram Bot API, DeepSeek Chat Completions, GitHub REST, and Google OAuth/Calendar REST.

## Global Constraints

- Support exactly one configured Telegram owner and exactly one `eggyd` replica.
- Keep provider SDK types, credentials, and errors out of `internal/kernel` and `internal/ports`.
- Use `/data/config.yaml`, `/data/state.json`, `/data/MEMORY.md`, and `/data/codex` in deployment.
- Require separate, expiring, payload-digest-bound approvals for commit, push, pull-request creation, and Calendar mutation.
- Reject direct pushes to configured protected branches even when approved; never merge or self-deploy.
- Use atomic, version-checked file writes and encrypt Google refresh tokens with `EGGY_ENCRYPTION_KEY`.
- Bound model tool turns, network retries, process time/output, proactive messages, and workspace retention.
- Default to standard-library implementations; use no web framework, ORM, DI framework, agent framework, native plugin runtime, or database.
